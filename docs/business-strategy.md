# Linoa 事業戦略ドキュメント

作成日: 2026-03-06

---

## 1. プロダクト概要

**Linoa** — 飲食店の「店舗専属AI秘書」

LINE公式アカウントを入口に、業務効率化からAI経営支援までを一気通貫で提供するプロダクト。単なる業務効率化ツールではなく、「分析→提案→実行支援」まで行うAI秘書として、既存SaaSとは異なるポジションを狙う。

### コアコンセプト

```
既存SaaS（Airメイト等）: データ → 分析 → 表示（ここで終わり）
Linoa:                   データ → 分析 → 提案 → 実行支援（ここまでやる）
```

「業務を楽にする」ではなく「売上を上げる秘書」。

---

## 2. ターゲット

- 店長 + バイト5人程度の小規模飲食店
- 業務効率化のメスは入れたことがあるが、目に見えて楽になっていない・売上が上がっている感じがしない層
- Airシフト等の専用SaaSを導入するほどではないが、LINEグループ+紙+Excelでの管理に限界を感じている店舗

## 3. 競合分析

### Airシリーズ（リクルート）

| サービス | 強み | 弱み |
|----------|------|------|
| Airレジ | 無料POSレジで圧倒的シェア。入口として最強 | APIがクローズド。エコシステム囲い込み |
| Airシフト | シフトボードDL数1,200万。バイト向けアプリ浸透 | シフト管理特化。月額330円/人 |
| Airメイト | 売上分析・原価率可視化 | **分析止まり。「だからどうする」がない** |

**Airシリーズの構造的弱点**: 「管理ツール」であって「コミュニケーション」がない。店長とバイトの日常連絡はLINEグループで行われているのが実態。

### SNS生成・集客支援

| サービス | 特徴 | 弱み |
|----------|------|------|
| マデカラエージェント（Mab） | 飲食店特化。SNS投稿・口コミ返信・競合分析。月額9,800円〜 | 集客特化。店舗運営（シフト・在庫等）は対象外。建築・設計がバックグラウンド |
| Canva AI | テンプレートからPOP・メニュー・SNS素材を生成 | 汎用ツール。飲食店のデータ文脈を知らない |

### フードロス・在庫管理

| サービス | 特徴 | 弱み |
|----------|------|------|
| DATAFLUCT「foodloss」 | POS+天候データで需要予測。動的価格調整 | チェーン向け。小規模店には重い |
| TABETE | 余剰食品を消費者にマッチング | 店内活用（メニュー変更等）ではなく外部販売 |
| NEC / 日立 | 大規模需要予測・自動発注 | エンタープライズ向け |

### 市場の空白地帯

**「データに基づいてSNS投稿やPOPを生成する」プレイヤーがいない。**

- マデカラ: SNS投稿を作る（在庫のことは知らない）
- Airメイト: 売上分析する（SNSのことは知らない）
- Canva: デザインを作る（何を訴求すべきかは人間が考える）

Linoaは全データを1つのAIが持ち、「貝が余ってる→おすすめメニュー変更→POP作成→Instagram投稿」が一気通貫で繋がる。

## 4. Linoaの差別化・勝ち筋

### Airシリーズとの関係

正面突破はしない。レイヤーが異なる。

```
Airシリーズ = 「管理の効率化」（ツール）
Linoa       = 「経営の意思決定支援」（パートナー）
```

将来的にはAirシリーズの上に乗るレイヤー、もしくはAirレジのデータと連携する形も視野に入れる。

### LINEを軸にする意味

- バイトに新しいアプリを入れさせる必要がない（全員がLINEを持っている）
- 「通知を受ける・簡単な入力をする」場所としてLINEは最適
- Webアプリは「じっくり見る・管理する」場所として役割分担

### 「秘書」と「ツール集」の決定的な違い

個々の機能は単体サービスとして作れるが、Linoaの強みは「全部が1つのAIの頭の中で繋がっている」こと。バラバラのSaaSを束ねたのではなく、1つのAIが全文脈を持って判断・提案・実行する。

## 5. アーキテクチャ思想

```
LINE公式アカウント = 業務の「入口」と「配信口」
  - データ入力（質疑応答形式）
  - 通知受信（シフト確定、賞味期限アラート、リマインダー等）
  - 写真送信（賞味期限、マニュアル素材等）

Webアプリ = 「管理・調整・閲覧」の場
  - ダッシュボード（売上・客数グラフ）
  - シフト調整・確定
  - コンテンツ確認・編集・承認
  - 各種設定

AI秘書エンジン = 「頭脳」
  - データ分析（売上傾向、原価率、客数変動）
  - 課題検知（賞味期限、売上低下、人手不足）
  - 施策提案（メニュー変更、クーポン、仕入れ調整）
  - コンテンツ生成（POP、SNS投稿、メニュー表）
```

### ロール別UI

| ロール | LINE側 | Web側 |
|--------|--------|-------|
| 店長 | 通知受信・AI対話・日報確認 | ダッシュボード・シフト調整・コンテンツ承認 |
| バイト | シフト提出・売上入力・所感入力・賞味期限写真 | （基本的に不要） |

## 6. 全機能マップ

### 売上を上げる系
- SNS投稿・ストーリー生成（データ連動）
- 店内POP・メニュー表作成
- クーポン・キャンペーン設計
- 口コミ返信の自動生成（Google Maps・食べログ）
- リピーター分析

### コストを下げる系
- 賞味期限管理→フードロス削減→メニュー提案
- 仕入れ最適化（天候×来客予測）
- 人件費シミュレーション（シフト×売上で適正人員提案）

### 人を動かす系
- シフト管理（収集→調整→配信）
- 新人教育（マニュアルbot）
- バイトへのタスク通知

### 経営判断を支える系
- 日報分析→週次/月次レポート
- 天候×売上の相関分析
- 競合店の動向モニタリング
- 税理士・会計士向けレポート出力

### 店の外とつなぐ系
- Uber Eats/出前館のメニュー最適化
- Googleビジネスプロフィール更新
- 予約管理

## 7. データ戦略

### 基本方針

将来のAI秘書に必要なデータから逆算して、今の機能（=業務効率化）を決める。

### データ分類

**A. ユーザーに入力してもらうデータ（業務効率化として価値を返す）**
- 日報（売上・客数・所感）
- 賞味期限（写真+テキスト）
- シフト
- メニュー構成

**B. 自動取得できるデータ（ユーザーの手間ゼロ）**
- 天候（外部API）
- 口コミ（Google API）
- SNS反応（Instagram API）

### データ蓄積→AI活用の段階

| 蓄積期間 | 可能になること |
|----------|--------------|
| 即日 | 店舗情報+所感→業態に合わせた一般的アドバイス。賞味期限→アラート |
| 2週間 | 日報7〜14件→曜日別傾向 |
| 1ヶ月 | 日報30件+天候→相関分析。シフト→人件費率可視化 |
| 3ヶ月 | 全データ蓄積→本格的なAI秘書（課題検知+施策提案+実行支援） |

## 8. POS連携戦略

### 各POSレジのAPI開放状況

| POSレジ | API公開 | 備考 |
|---------|---------|------|
| スマレジ | 開放（充実） | OAuth 2.0、Webhook対応。開発者ポータルあり。**最有力連携先** |
| Square | 開放（非常に充実） | ドキュメント豊富。日本の小規模飲食店での普及率はスマレジより低い |
| ユビレジ | 開放（プレミアムのみ） | OAuth 2.0。技術サポートは有償 |
| Airレジ | 限定的 | 連携先が指定システムのみ。自由な開発者向けAPIではない |
| USENレジ | 非公開 | CSV出力のみ |

### 方針

- MVPでは手入力。POS連携なしでもAI秘書の価値は出せる
- 中期でスマレジ→Square連携（手入力を自動化するアップグレード）
- Airレジがクローズドなことは逆にチャンス（Airレジユーザーへの分析サービスの選択肢がない）

## 9. 3年ロードマップ

### Year 1: 基盤構築 + 初期ユーザー獲得

**Q1（Month 1-3）: 業務効率化で入り込む**
- Month 1: ダッシュボード + リッチメニュー + 天候API自動取得
- Month 2: 賞味期限管理（写真+テキスト→アラート通知）
- Month 3: シフト管理（簡易版: 収集→一覧→確定配信）
- 目標: テスト店舗 3〜5店で実運用開始

**Q2（Month 4-6）: AI秘書が目覚める**
- Month 4: AI週次レポート配信（売上傾向・曜日分析・天候相関）
- Month 5: AI施策提案（食材余り→メニュー提案、客数低下→対策案）
- Month 6: 自然言語対話（「最近どう？」で状況サマリー返答）
- 目標: 有料プランβ提供開始、10店舗突破

**Q3（Month 7-9）: 提案→実行の一気通貫**
- Month 7: SNS投稿生成（Instagramストーリー / 投稿テキスト+画像）
- Month 8: 店内POP・メニュー表生成
- Month 9: コンテンツ生成エンジン統合
- 目標: 正式料金プラン確定、月額課金開始

**Q4（Month 10-12）: 磨き込み + 拡張**
- Month 10: 口コミ自動返信
- Month 11: 新人教育マニュアルbot
- Month 12: 予約管理（LINE経由）
- 目標: 30〜50店舗

### Year 2: スケール + データの壁を超える

**Q1（Month 13-15）: POS連携で手入力を解放**
- スマレジAPI連携、Square API連携
- 日報の売上・客数が自動入力に
- 商品別売上データでメニュー分析が本格化

**Q2（Month 16-18）: マルチ店舗対応**
- 本部ダッシュボード（複数店舗の横比較）
- 店舗間ベンチマーク
- ターゲット拡大: 個人店→3〜10店舗のミニチェーン

**Q3（Month 19-21）: 仕入れ最適化**
- 需要予測（過去データ+天候+イベント→来客数予測）
- 仕入れ量の自動提案
- 仕入れ業者との連携

**Q4（Month 22-24）: 会計・労務連携**
- freee / マネーフォワード連携
- 勤怠管理（シフト→実績→給与計算）
- 確定申告サポート
- 目標: 200〜500店舗、シリーズA水準の実績

### Year 3: プラットフォーム化

**Q1-Q2（Month 25-30）: 業界データの力**
- 匿名化した業界ベンチマーク提供
- 食材価格のトレンド分析
- 成功パターンのAI学習
- 店舗数が増えるほどデータ価値が上がるネットワーク効果

**Q3-Q4（Month 31-36）: エコシステム**
- Linoa API公開
- Airレジ連携（交渉）
- デリバリー連携（Uber Eats / 出前館）
- 不動産・出店支援（データに基づく出店エリア提案）
- 金融連携（売上データに基づく融資審査サポート）
- 目標: 1,000店舗超

### 3年間の構造

```
Year 1: 「便利なLINE Bot」→「AI秘書」への進化（手入力でデータを貯める）
Year 2: 「AI秘書」→「経営パートナー」への進化（POS連携で自動化、予測・最適化）
Year 3: 「経営パートナー」→「業界インフラ」への進化（集合データのネットワーク効果）
```

前のフェーズで集めたデータが次のフェーズの武器になる設計。

## 11. 法人化について

### タイミング

VCから出資を受ける予定があるなら早い方がいい。VCは個人に出資できず、株式会社への出資が前提。テスト店舗との契約を進める上でも法人格があった方が信用される。

### 費用

- 株式会社: 約18〜20万円（定款認証+登録免許税）
- 合同会社: 約6〜7万円（定款認証不要）

### 登記費用の調達手段

1. 合同会社で始めて後から株式会社に組織変更（6万円→追加6万円）
2. freee会社設立 / マネーフォワード会社設立で士業費用を削減
3. 日本政策金融公庫の新創業融資（個人でも申込可、審査1〜2ヶ月）
4. VCにブリッジで相談（J-KISS等で個人→法人化後に株式転換）
5. 創業支援助成金（東京都等、上限400万円）
